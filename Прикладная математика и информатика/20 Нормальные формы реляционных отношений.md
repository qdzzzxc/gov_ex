# Нормальные формы реляционных отношений. Краткая характеристика основных нормальных форм. Нормализация отношений. Применимость нормальных форм к прикладным задачам хранения данных.

## TL;DR

**Функциональная зависимость (ФЗ)** $X \to Y$: значения $X$ однозначно определяют $Y$ (внутри одной таблицы). Это бизнес-правило, не свойство конкретных данных. Не путать со связями между таблицами (1:1, 1:N) — это разное.

**Аномалии** возникают из-за дублирования в плохо спроектированной таблице:

- **Обновления**: меняем телефон клиента → надо во всех 1000 его заказах.
- **Вставки**: не можем сохранить отдел без сотрудника, если они в одной таблице.
- **Удаления**: удалили последний заказ клиента → потеряли его телефон.

**Цель нормализации:** избежать аномалий через разбиение таблиц.

### Нормальные формы

**1НФ — атомарность.** Каждая ячейка — одно скалярное значение, не список.

Нарушение: ячейка `телефоны = "111-22, 333-44"`.

**2НФ — нет частичных зависимостей от составного ключа.** Применима только при составном ключе. Неключевой атрибут не должен зависеть от **части** ключа.

Нарушение: `ЗаказПозиция(№заказа, товар, количество, цена_товара)` с ключом `(№заказа, товар)`. ФЗ `товар → цена` — частичная. Чинится выносом `Товар(товар, цена)` в отдельную таблицу.

**3НФ — нет транзитивных зависимостей.** Неключевой атрибут не должен зависеть от ключа через другой неключевой.

Нарушение: `Сотрудник(id, имя, отдел, начальник_отдела)`. Цепочка `id → отдел → начальник_отдела`. Чинится выносом `Отдел(отдел, начальник_отдела)`.

Это **рабочий уровень** для большинства реальных OLTP-баз.

**BCNF (Бойса-Кодда).** Усиленная 3НФ: каждый детерминант ФЗ должен быть суперключом. Покрывает редкие случаи с пересекающимися ключами-кандидатами. Не всегда сохраняет ФЗ при декомпозиции.

**4НФ.** Запрещает многозначные зависимости — независимые "много-ко-многим" в одной таблице (сотрудник × навыки × проекты не должны быть вместе).

**5НФ.** Сложные случаи, когда таблицу нужно разбить на 3+ и потом восстанавливать через JOIN.

**6НФ.** Каждая таблица содержит ровно один неключевой атрибут. Используется в темпоральных БД.

**Примеры:**
- BCNF: пересекающиеся ключи-кандидаты, ФЗ от не-суперключа (часто связано с ролями типа `(студент, предмет) ↔ (студент, преподаватель)`).
- 4НФ: независимые многозначные атрибуты у одной сущности (навыки × языки сотрудника).
- 5НФ: тройные связи, не разбиваемые на две таблицы (агент × компания × продукт).

### Декомпозиция — два требования

- **Lossless join**: после JOIN-а разбитых таблиц получаем исходную без потерь.
- **Сохранение ФЗ**: исходные зависимости выводимы из новых таблиц.

Иногда нельзя обеспечить оба → останавливаются на 3НФ вместо BCNF.

### Применимость

**OLTP** (транзакционные системы — банки, магазины): нормализация 3НФ/BCNF, целостность важнее скорости чтения.

**OLAP / хранилища данных**: денормализация ради скорости. Схемы:
- **Звезда** — таблица фактов + плоские денормализованные таблицы измерений.
- **Снежинка** — звезда, но измерения нормализованы в иерархию (меньше дублирования, больше JOIN-ов).

**NoSQL**: классические нормальные формы не применяются. Денормализация по умолчанию, моделирование под запросы. Целостность поддерживается приложением. Embedded (вложенность) vs referenced (по id) — прикладной выбор.

### Главный принцип

> Реляционка нормализуется по умолчанию (и денормализуется точечно для скорости). OLAP/NoSQL денормализуются по умолчанию (и нормализуются точечно для целостности).
> 
> Выбор уровня нормализации — компромисс между целостностью записи и скоростью чтения. Не самоцель.
