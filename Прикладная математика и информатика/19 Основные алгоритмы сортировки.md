# Основные алгоритмы сортировки. Их сложность, достоинства и недостатки. Алгоритм быстрого поиска в отсортированном массиве. Сложность поиска в отсортированном и не отсортированном массиве.

## Критерии сравнения
Время (среднее/худшее/лучшее), память (in-place или нет), стабильность — сохранение порядка равных элементов, нужна для многоэтапной сортировки по нескольким полям.
## Простые сортировки $O(n^2)$
- Пузырёк: меняем соседей. Стабилен. С early-exit лучшее $O(n)$ на отсортированном.
- Вставки: проталкиваем элемент в отсортированный префикс. Стабилен. На почти отсортированном $O(n)$.
- Выбор: ищем минимум в остатке, ставим в начало. Не стабилен. Всегда $O(n^2)$ независимо от данных.
## Эффективные сортировки $O(n \log n)$
- Quicksort: делим по pivot на меньшие/большие, рекурсия. Средне $O(n \log n)$, худшее $O(n^2)$ при плохом pivot. In-place, не стабилен (при разделении равные элементы могут перепрыгнуть друг друга). Защита от худшего случая — random pivot или introsort.
- Mergesort: делим пополам, сортируем рекурсивно, сливаем. Всегда $O(n \log n)$, память $O(n)$ на слияние, стабилен.
- Heapsort: строим max-heap из массива ($O(n)$), $n$ раз извлекаем максимум в конец. Всегда $O(n \log n)$, in-place, не стабилен. Даёт гарантию там, где quicksort может выродиться.
- Timsort: гибрид merge + insertion, находит уже отсортированные runs и сливает. Реализация Python `sorted`. Стабилен, $O(n)$ на почти отсортированных.
## Heap (структура)
Бинарное дерево со свойством родитель ≥ детей (max-heap), хранится в массиве: дети узла $i$ на индексах $2i+1$, $2i+2$ (0-индексация). Insert и extract-max — $O(\log n)$ через sift up/down. Используется как priority queue.
## Линейные сортировки $O(n + k)$
Counting, radix, bucket — обходят $\Omega(n \log n)$, используя структуру ключей (целые в ограниченном диапазоне). Counting считает частоты, radix — поразрядно через counting, bucket распределяет по диапазонам. При большом $k$ counting хуже $n \log n$ — линейность только при $k = O(n)$.
## Нижняя граница
Сравнительные сортировки требуют $\Omega(n \log n)$. Доказательство через дерево решений: $n!$ возможных перестановок, дерево с бинарными ветвлениями имеет высоту $\ge \log_2 n! \sim n \log n$ по Стирлингу.
## Поиск в неотсортированном массиве
Только линейный, $O(n)$. Быстрее нельзя — без структуры данных любая невидимая позиция может содержать искомый элемент.
## Бинарный поиск в отсортированном
$O(\log n)$ время, $O(1)$ память. Требует массив (случайный доступ за $O(1)$) и упорядоченность.
```python
def bsearch(a, x):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        m = (lo + hi) // 2
        if a[m] == x: return m
        elif a[m] < x: lo = m + 1
        else: hi = m - 1
    return -1
```
Линейный vs бинарный: один поиск — линейный, сортировка не окупится. Много поисков — отсортировать + бинарный, окупается при $k \gtrsim \log n$.
## Бинпоиск по предикату
Паттерн «минимальное значение $X$, при котором выполнено условие». Определяешь `can(X)`, бинпоишь по $X$. Работает при монотонности предиката (F F F T T T). Примеры: $\lfloor\sqrt{x}\rfloor$, минимальная грузоподъёмность корабля.
