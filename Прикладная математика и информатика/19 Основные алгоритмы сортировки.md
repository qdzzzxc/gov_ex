# Основные алгоритмы сортировки. Их сложность, достоинства и недостатки. Алгоритм быстрого поиска в отсортированном массиве. Сложность поиска в отсортированном и не отсортированном массиве.

## TL;DR

**Критерии сравнения:** время (среднее/худшее/лучшее), память (in-place или нет), стабильность (сохранение порядка равных элементов — нужна для многоэтапной сортировки по нескольким полям).

**Простые ($O(n^2)$):**
- **Пузырёк:** меняем соседей. Стабилен.
- **Вставки:** проталкиваем элемент в отсортированный префикс ("куда поставить это?"). Стабилен. На почти отсортированном — $O(n)$.
- **Выбор:** ищем минимум в остатке, ставим в начало ("какое число поставить сюда?"). Не стабилен. Всегда $O(n^2)$ независимо от данных.

**Эффективные ($O(n \log n)$):**
- **Quicksort:** делим по pivot на меньшие/большие, рекурсия. Средне $O(n \log n)$, худшее $O(n^2)$ (плохой pivot на отсортированном массиве). In-place, не стабилен. Защита: random pivot, median-of-three, introsort.
- **Mergesort:** делим пополам, сортируем рекурсивно, сливаем. Всегда $O(n \log n)$, память $O(n)$ на слияние, стабилен.
- **Heapsort:** строим max-heap из массива ($O(n)$), $n$ раз извлекаем максимум в конец. Всегда $O(n \log n)$, in-place, не стабилен. Дает гарантию там, где quicksort может выродиться.
- **Timsort:** гибрид merge + insertion, находит уже отсортированные runs и сливает. В Python `sorted`, Java `Arrays.sort`. Стабилен, $O(n)$ на почти отсортированных.

**Heap (структура):** бинарное дерево со свойством родитель ≥ детей (max-heap), хранится в массиве: дети узла `i` на индексах `2i+1`, `2i+2`. Insert и extract-max — $O(\log n)$ через sift up/down. Используется как priority queue (Дейкстра, A*, top-K, планировщики).

**Линейные ($O(n + k)$):** counting, radix, bucket — обходят $\Omega(n \log n)$, используя структуру ключей (целые в ограниченном диапазоне). Counting считает частоты, radix — поразрядно через counting, bucket распределяет по диапазонам.

**Нижняя граница** сравнительных сортировок: $\Omega(n \log n)$ через дерево решений (высота $\ge \log_2 n!$).

**Поиск в неотсортированном массиве:** только линейный, $O(n)$. Быстрее нельзя — без структуры данных любая невидимая позиция может содержать искомый элемент.

**Бинарный поиск в отсортированном:** $O(\log n)$, $O(1)$ память. Требует массив (случайный доступ за $O(1)$) и упорядоченность.

```python
def bsearch(a, x):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        m = (lo + hi) // 2
        if a[m] == x: return m
        elif a[m] < x: lo = m + 1
        else: hi = m - 1
    return -1
```

**Линейный vs бинарный:** один поиск → линейный (сортировка не окупится). Много поисков → отсортировать + бинарный, окупается при $k \gtrsim \log n$.

**Бинпоиск на монотонном предикате** — паттерн "минимальное значение $X$, при котором выполнено условие". Определяешь `can(X)` (функцию проверки), бинпоишь по $X$. Работает, если предикат монотонен (F F F T T T). Примеры: $\lfloor\sqrt{x}\rfloor$, минимальная грузоподъёмность корабля.

**Альтернативы бинпоиску для многократного поиска:**

- Хеш-таблица: $O(1)$ в среднем, порядок не сохраняется.
- Сбалансированное дерево: $O(\log n)$ на всё, порядок есть, поиск диапазонов и ближайшего.
- Trie: $O(L)$ для строк длины $L$.
