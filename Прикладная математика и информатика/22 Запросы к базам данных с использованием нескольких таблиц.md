# Запросы к базам данных с использованием нескольких таблиц. Объединение и пересечение отношений. Соединения отношений — основные виды соединений, их характеристики.

## TL;DR
Соединения объединяют строки из разных таблиц. **JOIN** — по условию `ON`: `INNER` (только совпадения), `LEFT/RIGHT/FULL OUTER` (все строки одной/обеих + `NULL` для несовпадений), `CROSS` (декартово произведение), `SELF` (с самой собой), `NATURAL` (по одноимённым столбцам, опасно). **Множественные операции** — `UNION` (объединение, дедуп), `UNION ALL` (без дедупа), `INTERSECT` (пересечение), `EXCEPT` (разность). Альтернативы соединению — подзапросы, `EXISTS`, `IN`, CTE. Выбор — по семантике задачи и оптимизатору.

## Развёрнуто

### Виды соединений

**`INNER JOIN ... ON cond`** — соединение строк по условию: оставляет только пары строк, удовлетворяющие `cond`, и объединяет их столбцы. Это не то же самое, что множественное пересечение `INTERSECT`.
```sql
SELECT e.name, d.title
FROM employees e
INNER JOIN departments d ON e.dep_id = d.id;
```

**`LEFT OUTER JOIN`** — все строки левой таблицы; для несовпадающих — `NULL` в столбцах правой. Полезно для «найти X и присоединить Y, если есть».

**`RIGHT OUTER JOIN`** — то же, наоборот. Реже используется (можно поменять таблицы местами).

**`FULL OUTER JOIN`** — все строки из обеих, несовпадающие пары — с `NULL`. Не во всех СУБД (в MySQL эмулируется через `UNION` LEFT и RIGHT).

**`CROSS JOIN`** — декартово произведение: каждая с каждой. Без `ON`. Случайный декартов результат обычно возникает при `CROSS JOIN` или старой записи через запятую без условия соединения.

**`SELF JOIN`** — таблица с самой собой (по разным алиасам). Например, иерархии:
```sql
SELECT e.name, m.name AS manager
FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;
```

**`NATURAL JOIN`** — соединение по всем одноимённым столбцам. Опасно: добавление столбца может изменить семантику запроса. Лучше `JOIN ... USING(col)` или явное `ON`.

**Эквисоединение** vs **тета-соединение**: первое — `=` в условии, второе — произвольный предикат (`<, >, BETWEEN`).

### Множественные операции (Union/Intersect/Except)
Применяются к двум `SELECT` с одинаковой структурой (число и типы столбцов).

- **`UNION`** — объединение с устранением дубликатов.
- **`UNION ALL`** — объединение без дедупликации (быстрее).
- **`INTERSECT`** — пересечение (только общие строки).
- **`EXCEPT`** (или `MINUS` в Oracle) — разность (строки первого, не входящие во второй).

```sql
SELECT customer_id FROM orders_2024
UNION
SELECT customer_id FROM orders_2025;
```

### Подзапросы
- **Скалярный** (возвращает одно значение):
```sql
SELECT name FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);
```
- **В `FROM`** (производная таблица): `FROM (SELECT ... ) AS t`.
- **В `WHERE`** через `IN`/`NOT IN`/`ANY`/`ALL`:
```sql
SELECT * FROM products WHERE id IN (SELECT product_id FROM sales);
```
- **`EXISTS`/`NOT EXISTS`** — проверка наличия строк:
```sql
SELECT * FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id);
```
- **Коррелированные подзапросы** — внутренний запрос ссылается на внешний; выполняется для каждой строки внешнего.

### CTE (Common Table Expression, `WITH`)
```sql
WITH dept_avg AS (
    SELECT dep_id, AVG(salary) AS avg_sal
    FROM employees GROUP BY dep_id
)
SELECT e.name, e.salary, d.avg_sal
FROM employees e JOIN dept_avg d ON e.dep_id = d.dep_id
WHERE e.salary > d.avg_sal;
```
**Рекурсивные CTE** — для иерархий и графов:
```sql
WITH RECURSIVE chain(id, mgr) AS (
    SELECT id, manager_id FROM employees WHERE id = 42
    UNION ALL
    SELECT e.id, e.manager_id
    FROM employees e JOIN chain c ON e.id = c.mgr
)
SELECT * FROM chain;
```

### Когда что использовать
- Нужны столбцы из обеих таблиц → **JOIN**.
- Нужно лишь проверить наличие → **EXISTS** (часто эффективнее `IN`).
- Списки фиксированных значений → **IN (...)**.
- Объединение результатов разных запросов → **UNION**.
- Структурный (повторно используемый) шаг → **CTE**.

### Пример с тремя таблицами
```sql
SELECT s.name, c.title, g.grade
FROM students s
JOIN grades   g ON g.student_id = s.id
JOIN courses  c ON c.id = g.course_id
WHERE g.grade >= 4
ORDER BY s.name, c.title;
```

### Подводные камни
- **Случайное декартово произведение** — использовали `CROSS JOIN` или comma join без предиката соединения, объём результата $\times$.
- **`OUTER JOIN` + `WHERE` на правой таблице**: фильтр в `WHERE` отбрасывает `NULL`-строки, превращая `LEFT JOIN` в `INNER`. Решение: переносить условие в `ON`.
- **`NOT IN` с `NULL`** в подзапросе: возвращает пустой результат, потому что любое сравнение с `NULL` — `UNKNOWN`. Использовать `NOT EXISTS`.
- **`UNION` дедуплицирует** (обычно через сортировку или хеширование, поэтому медленнее). Порядок результата всё равно не определён без `ORDER BY`; если дубликаты невозможны или не страшны — `UNION ALL`.
- **`NATURAL JOIN`** хрупок к добавлению столбцов и плохо читается.
- На больших таблицах: оптимизатор может выбрать разные планы (hash-join, merge-join, nested loop) — стоит проверять `EXPLAIN`.
