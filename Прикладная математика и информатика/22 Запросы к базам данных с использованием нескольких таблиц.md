# Запросы к базам данных с использованием нескольких таблиц. Объединение и пересечение отношений. Соединения отношений — основные виды соединений, их характеристики.

## TL;DR
Соединения объединяют строки из разных таблиц. **JOIN** работает по условию `ON`: `INNER` оставляет только совпадения, `LEFT/RIGHT/FULL OUTER` сохраняют строки одной или обеих таблиц с `NULL` для несовпадений, `CROSS` даёт декартово произведение, `SELF` соединяет таблицу с самой собой. Множественные операции работают с результатами `SELECT`: `UNION`, `UNION ALL`, `INTERSECT`, `EXCEPT`. Подзапросы, `EXISTS`, `IN` и CTE нужны, когда удобнее выразить условие или промежуточный шаг.

## JOIN-ы
**`INNER JOIN ... ON cond`** — оставляет только пары строк, удовлетворяющие условию соединения.

```sql
SELECT e.name, d.title
FROM employees e
JOIN departments d ON e.dep_id = d.id;
```

**`LEFT OUTER JOIN`** — все строки левой таблицы плюс совпадения справа; если совпадения нет, справа будут `NULL`. Полезно для задач "показать всех клиентов и их заказы, если заказы есть".

**`RIGHT OUTER JOIN`** — зеркальный вариант `LEFT JOIN`. Используется реже, потому что обычно можно поменять таблицы местами.

**`FULL OUTER JOIN`** — все строки обеих таблиц: совпавшие объединяются, несовпавшие дополняются `NULL`. Не во всех СУБД доступен напрямую; в MySQL часто эмулируется через `UNION` двух outer join.

**`CROSS JOIN`** — декартово произведение $n \times m$, без условия `ON`. Иногда нужен для генерации всех комбинаций, но случайно возникает, если забыть условие соединения.

**`SELF JOIN`** — соединение таблицы с самой собой через разные алиасы. Типичный пример — иерархия сотрудников:

```sql
SELECT e.name, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
```

**`NATURAL JOIN`** соединяет по всем одноимённым столбцам автоматически. Опасен: добавление нового одноимённого столбца может незаметно изменить смысл запроса. На экзамене лучше говорить, что предпочтительны явные `ON` или `USING`.

**Эквисоединение** — условие через `=`. **Тета-соединение** — условие через произвольный предикат: `<`, `>`, `BETWEEN`, пересечение интервалов и т.п.

## Множественные операции
Операции применяются к двум `SELECT` с одинаковым числом столбцов и совместимыми типами. Имена столбцов обычно берутся из первого запроса.

- **`UNION`** — объединение с устранением дубликатов.
- **`UNION ALL`** — объединение без устранения дубликатов, быстрее.
- **`INTERSECT`** — строки, присутствующие в обоих результатах.
- **`EXCEPT`** или `MINUS` в Oracle — строки первого результата, которых нет во втором.

```sql
SELECT customer_id FROM orders_2024
UNION
SELECT customer_id FROM orders_2025;
```

`JOIN` и `INTERSECT` не одно и то же: `JOIN` склеивает столбцы строк по условию, а `INTERSECT` ищет одинаковые строки в двух результатах.

## Подзапросы
**Скалярный подзапрос** возвращает одно значение:

```sql
SELECT name
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
```

**Подзапрос в `FROM`** создаёт производную таблицу:

```sql
SELECT dep_id, avg_sal
FROM (
    SELECT dep_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY dep_id
) t;
```

**`IN` / `NOT IN`** проверяют вхождение в список значений. **`EXISTS` / `NOT EXISTS`** проверяют наличие строк и часто лучше подходят для коррелированных подзапросов:

```sql
SELECT *
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

Коррелированный подзапрос ссылается на столбцы внешнего запроса и логически выполняется для каждой строки внешнего результата. Оптимизатор может переписать его в join, но семантически это "проверка относительно текущей строки".

## CTE (`WITH`)
CTE — именованный временный результат внутри одного запроса. Он делает запрос читаемее и позволяет переиспользовать промежуточный шаг.

```sql
WITH dept_avg AS (
    SELECT dep_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY dep_id
)
SELECT e.name, e.salary, d.avg_sal
FROM employees e
JOIN dept_avg d ON e.dep_id = d.dep_id
WHERE e.salary > d.avg_sal;
```

Рекурсивные CTE (`WITH RECURSIVE`) применяются для иерархий и графов.

## Пример с несколькими таблицами
```sql
SELECT s.name, c.title, g.grade
FROM students s
JOIN grades  g ON g.student_id = s.id
JOIN courses c ON c.id = g.course_id
WHERE g.grade >= 4
ORDER BY s.name, c.title;
```

## Когда что использовать
- Нужны столбцы из обеих таблиц — `JOIN`.
- Нужно проверить наличие связанных строк — `EXISTS`.
- Нужно сравнить со списком значений — `IN`.
- Нужно объединить однотипные результаты — `UNION` / `UNION ALL`.
- Нужно оформить промежуточный шаг — CTE.

## Подводные камни
- Случайное декартово произведение: забыли `ON` или написали comma join без условия, получили $n \times m$ строк.
- `LEFT JOIN` плюс фильтр по правой таблице в `WHERE` превращает результат в `INNER JOIN`, потому что `NULL`-строки отбрасываются. Условие для правой таблицы часто надо переносить в `ON`.
- `NOT IN` с `NULL` в подзапросе может вернуть пустой результат из-за трёхзначной логики SQL. Надёжнее использовать `NOT EXISTS`.
- `UNION` дедуплицирует и поэтому обычно медленнее `UNION ALL`.
- `NATURAL JOIN` хрупок к изменению схемы.
- Без `ORDER BY` порядок строк не гарантирован.

## Главное на экзамене
1. Назвать `INNER`, `LEFT`, `RIGHT`, `FULL`, `CROSS`, `SELF JOIN`.
2. Объяснить разницу `JOIN` и множественных операций.
3. Запомнить требования `UNION/INTERSECT/EXCEPT`: одинаковое число столбцов и совместимые типы.
4. Объяснить `EXISTS` vs `IN`, особенно проблему `NOT IN` и `NULL`.
5. Объяснить CTE как именованный промежуточный результат.
