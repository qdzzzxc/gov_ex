# Подготовка к ГИА — конвенции для ответов

Проект — Obsidian-vault для подготовки к госэкзамену (28 мая 2026).
- **45 теоретических вопросов**, разбитых на два блока: «Прикладная математика и информатика» (25) и «Прикладное машинное обучение» (20). Каждый вопрос — отдельный `.md`-файл.
- Из профильного блока «Прикладное машинное обучение» исключены вопросы 6-10: модели индивидуального иска; договоры страхования с вычетом, лимитом и франшизой; индивидуальные модели риска; коллективные модели риска; эмпирическое распределение / огива / Нельсон-Аален.
- **38 практико-ориентированных задач**, сгруппированных в 10 тем (алгоритмы и структуры данных, эконометрика, технологии обработки данных, численные методы, машинное обучение, финансовая математика, ТВ и матстат, ДУ, исследование операций, страхование). Каждая задача — тоже отдельный `.md`-файл.

Цель — заполнить все файлы однотипными, структурированными ответами, пригодными для подготовки и устного ответа на экзамене.

## Структура репозитория
- `Прикладная математика и информатика/` — 25 вопросов.
- `Прикладное машинное обучение/` — 20 вопросов: исходные вопросы 1-5 и 11-25.
- `Практико-ориентированные задания/<тема>/` — 38 задач в 10 подпапках.
- `_index.md` — оглавление всего проекта с чекбоксами `- [ ]` для всех 45+38 пунктов; единственное место, откуда видно общий прогресс и где удобно открывать файлы по wiki-ссылкам в Obsidian. По мере готовности ответа отмечаем `- [x]`.
- `sources/` — исходники: официальный PDF со списком вопросов (`Программа ГИА выпуск 2022 ПМ.pdf`) и скриншоты условий практических задач из методички.
- `.venv/` — Python-окружение для прогона кода в задачах (numpy/scipy/matplotlib/pandas/sympy).

## Контекст
- Дедлайн: экзамен 28 мая 2026.
- Среда: проект в формате Obsidian-vault (wiki-ссылки `[[...]]` и LaTeX в `$...$` / `$$...$$`).
- Источники: пишем с нуля, используя веб-поиск для уточнения формулировок, формул и фактов. PDF в `sources/` содержит только список вопросов — за содержанием туда не лезем.
- Темп: один вопрос/задача за подход, без спешки.

## Структура файла теоретического вопроса

```markdown
# <Полная формулировка вопроса — уже стоит в файле, не трогаем>

## TL;DR
<5–10 строк: суть для устного ответа. Только определение + ключевые формулы/идеи.>

## Развёрнуто
<Основной конспект. Адаптивная структура — см. ниже.>
```

## Структура файла практической задачи

```markdown
# Задача N. <короткое название>.

## Условие
<текст условия — переписан с PDF или со скриншота из sources/>

## Решение
### Идея
### Код              (если нужен прогон — используем .venv)
### Прогон           (вывод скрипта)
### Ответ
### Замечания        (опционально: интерпретация, чувствительность, типичные ошибки)
```

### Адаптивные секции в «Развёрнуто»
Обязательны:
- **Определение / постановка задачи**
- **Ключевые формулы и теоремы** — с условиями применимости
- **Пример** — числовой или схематичный, минимально достаточный

Добавляем только если по теме реально есть содержание:
- **Вывод / доказательство** — если упомянуто в формулировке вопроса или критично для понимания
- **Свойства / классификация**
- **Подводные камни / типичные ошибки**

Не плодим разделы ради формы. Лучше 3 живых секции, чем 6 пустых.

## Стиль

- **Язык**: русский. При первом упоминании англоязычного термина — англ. в скобках: «автокорреляция (autocorrelation)».
- **Формулы**: LaTeX. Inline: `$R^2$`. Блочно: `$$\hat\beta = (X^\top X)^{-1} X^\top y$$`.
- **Код** (Python / SQL / псевдокод): минимальные сниппеты, только если реально иллюстрируют идею. Не вставляем «полные рабочие примеры» ради объёма.
- **Объём «Развёрнуто»**: ориентир 1–3 экранных страницы. Если получается больше — значит, тема действительно того стоит.
- **Кросс-ссылки** `[[...]]` — только когда соседний вопрос реально нужен для понимания (напр. №13 «автокорреляция» → `[[12 ...МНК...]]`). Без фанатизма, не для красоты.

## Имена файлов и заголовки
- Имя файла — короткое (как сейчас), не трогаем.
- H1 внутри файла — полная формулировка из программы ГИА / условие задачи (уже стоит, не переписываем).
- `_index.md` — единое оглавление; добавление/удаление файлов отражаем там же. Чекбоксы `- [ ]` отмечаем по мере готовности.

## Рабочий процесс
1. Один вопрос/задача за заход. Не пакетим по 5 штук «костяком».
2. Перед написанием — короткий веб-поиск по формулировке, чтобы зацепить актуальные определения и стандартные обозначения. Не копипастим — пересказываем.
3. Для задач с кодом — прогон через `.venv`, вывод вставляем в раздел «Прогон».
4. После написания — пользователь читает, при необходимости просит углубить. Третий уровень детализации добавляем только точечно по запросу.
5. Когда ответ принят — отмечаем `- [x]` в `_index.md`.

## Чего не делаем
- Не пишем «введение» / «заключение» / «список литературы» — это не диплом.
- Не дублируем TL;DR в начале «Развёрнуто».
- Не вставляем картинки/диаграммы (Obsidian-плагины у пользователя не уточнены) — описываем словами или ASCII-схемой, если нужно.
- Не выдумываем факты. Если по теме что-то не помним точно — гуглим, а не сочиняем.
