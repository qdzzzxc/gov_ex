# Модели параллельного программирования и их сочетаемость с архитектурами параллельных вычислительных систем. Подходы к декомпозиции крупных вычислительных задач на подзадачи для параллельного исполнения.

## TL;DR
**Архитектуры**: SISD/SIMD/MISD/MIMD (Флинн), shared-memory (SMP, NUMA), distributed-memory (кластеры), GPU (SIMT). **Модели программирования**: shared-memory с потоками (OpenMP, pthreads, std::thread), message-passing (MPI), data-parallel (CUDA/OpenCL/SIMD), task-parallel (Cilk, TBB, Dask, Ray), MapReduce/Spark для big data. Соответствие: SMP→OpenMP/pthreads; кластер→MPI; GPU→CUDA/OpenCL; «облако данных»→Spark/Dask. **Декомпозиция**: данных (data parallel — одинаковая операция над разными частями), функциональная (task parallel — разные операции), pipeline (потоковая обработка). Ограничения — закон Амдаля и Густафсона.

## Развёрнуто

### Классификация архитектур (Флинн)
- **SISD**: один поток инструкций над одним потоком данных — классические CPU.
- **SIMD**: одна инструкция, много данных — векторные расширения (SSE, AVX), GPU (точнее — SIMT).
- **MISD**: редкая теоретическая категория (отказоустойчивые системы).
- **MIMD**: многоинструкционные многопотоковые — современные многоядерные CPU и кластеры.

### По организации памяти
- **Shared memory (SMP)**: процессоры разделяют общую RAM, обращение симметричное. Программируется через потоки. Ограничено несколькими десятками ядер.
- **NUMA (Non-Uniform Memory Access)**: разделённая память, но каждый узел имеет «свою» — обращение к чужой дороже. Большие серверные системы.
- **Distributed memory**: каждый узел со своей памятью, обмен — по сети. Кластеры, суперкомпьютеры. Программируются через MPI.
- **GPU (SIMT)**: тысячи лёгких потоков, исполняющих одинаковую программу над разными данными. Иерархия памяти (global, shared, registers).

### Модели параллельного программирования

**1. Многопоточная (shared memory)**:
- **OpenMP** — директивы препроцессора (`#pragma omp parallel for`), удобно для loop-parallelism.
- **pthreads / std::thread** — низкоуровневые потоки, ручная синхронизация (mutex, atomic, condvar).
- **Intel TBB / std::execution** — task-based, work stealing.
- Плюсы: лёгкое разделение данных. Минусы: race conditions, ложное разделение строк кэша (false sharing), ограничение по числу ядер.

**2. Передача сообщений (message passing)**:
- **MPI** — стандарт для кластеров. Процессы независимы, обмен через `MPI_Send/Recv`, коллективы `Bcast/Gather/Scatter/Allreduce`.
- Подходит для distributed memory; масштабируется до десятков тысяч узлов.
- Плюсы: явный контроль, никаких race conditions внутри процесса. Минусы: явная разработка коммуникации, отладка сложнее.

**3. Data-parallel / SIMD**:
- **CUDA / HIP / OpenCL** — GPU-вычисления.
- **SIMD-инструкции** CPU (intrinsics, autovectorization).
- Подходит для регулярных задач: BLAS, свёртки, симуляции на сетках, ML.
- Плюсы: огромная пропускная способность. Минусы: только при правильной локальности данных и регулярных вычислениях.

**4. Task-parallel / dataflow**:
- **Cilk Plus**, **OpenMP tasks**, **Intel TBB**.
- **Dask, Ray** — Python-фреймворки task-graph над кластером.
- Удобно для нерегулярных задач (рекурсии, балансирующиеся deque).

**5. MapReduce и data-flow для big data**:
- **Hadoop MapReduce** — простая модель: map → shuffle → reduce.
- **Apache Spark** — RDD/DataFrame, lazy DAG, in-memory. На порядки быстрее Hadoop.
- **Dask** — Python-аналог Spark, dataframe и bag.

**6. Acceleration / гибридные**:
- MPI + OpenMP — гибридная параллелизация на кластерах с многоядерными узлами.
- MPI + CUDA — распределённые GPU-вычисления (PyTorch DDP, Horovod).

### Соответствие моделей и архитектур
| Архитектура | Подходящая модель |
|---|---|
| SMP/многоядерный CPU | OpenMP, std::thread, TBB |
| NUMA-сервер | OpenMP с pinning, MPI на сокетах |
| Кластер | MPI |
| GPU | CUDA, OpenCL, OpenACC, frameworks (PyTorch/TF) |
| Big data cluster | Spark, Dask, Hadoop |
| Гетерогенный | MPI+OpenMP+CUDA, OpenACC |

### Подходы к декомпозиции

**1. Декомпозиция данных (data parallel)**: одна и та же операция применяется к разным частям набора данных. Пример: матричное умножение, обработка изображений по пикселям. Хорошо ложится на SIMD/SIMT.

**2. Функциональная декомпозиция (task parallel)**: разные процессы выполняют разные функции над общими или разными данными. Пример: pipeline обработки видео — декодирование, обработка, кодирование на разных ядрах.

**3. Pipeline (потоковая обработка)**: задача разбивается на стадии; данные «текут» через них. Пример: instruction pipelining в CPU; ETL-задачи.

**4. Master/Worker**: один процесс раздаёт задания, остальные выполняют. Подходит для динамической балансировки, нерегулярных задач.

**5. Domain decomposition**: расчётная область делится на подобласти (геометрически), на каждой — свой процесс с обменом по границам. Стандарт для PDE-симуляций.

### Закон Амдаля и Густафсона
**Амдаль**: ускорение фиксированной задачи на $p$ процессорах:
$$S(p) = \frac{1}{(1-f) + f/p},$$
$f$ — параллелизуемая доля. При $p\to\infty$, $S\to 1/(1-f)$ — плато из-за непараллелизуемой части.

**Густафсон**: «слабое масштабирование» — задача растёт с числом процессоров, $S(p) = (1-f) + p\cdot f$ — линейно.

На практике эффективность ограничена коммуникацией, синхронизацией, балансировкой нагрузки, гранулярностью задач.

### Метрики эффективности
- **Speedup** $S(p) = T_1/T_p$.
- **Efficiency** $E(p) = S(p)/p$.
- **Scalability**: как растёт $S$ с увеличением $p$ при фиксированной задаче (strong) или росте задачи (weak).

### Примеры
- **ML-обучение нейросети** на GPU: data parallel (батчи на разных GPU, all-reduce градиентов) + tensor parallel + pipeline parallel.
- **Поиск Гугла**: MapReduce/Spark — обработка инвертированного индекса.
- **CFD-симуляция**: domain decomposition + MPI + OpenMP внутри узла.

### Подводные камни
- Грубая гранулярность даёт плохую балансировку, мелкая — высокий overhead.
- Кеш-эффекты и NUMA: «горячая» память должна быть локальной.
- Race conditions, deadlocks — самые трудно-отлаживаемые баги в параллельном коде.
- При плохой архитектуре кода MPI может тормозить из-за коммуникации больше, чем параллелит.
